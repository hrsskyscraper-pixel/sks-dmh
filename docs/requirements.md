# 要件定義書 — できました表

## 1. システム概要

### 1.1 目的
新入社員（社員・メイト）のオンボーディング進捗を可視化・管理するウェブアプリケーション。
スキルの習得申請・認定フローをデジタル化し、マネージャーが担当メンバーの成長状況をリアルタイムで把握できるようにする。

### 1.2 技術スタック
- **フロントエンド**: Next.js (App Router)、Tailwind CSS、shadcn/ui
- **バックエンド**: Supabase（PostgreSQL + RLS + Storage + Auth）
- **認証**: Google OAuth 2.0
- **デプロイ**: Vercel

---

## 2. ユーザー種別と権限

| ロール | 名称 | 主な権限 |
|--------|------|---------|
| `employee` | 社員・メイト | 自分のスキル申請・確認のみ |
| `manager` | マネージャー | 担当チームのスキル認定・チーム変更申請 |
| `admin` | 開発者 | 全操作（システム管理含む） |
| `ops_manager` | 運用管理者 | 全操作（プロジェクト・チーム承認） |

### 2.1 雇用形態
- **社員** / **メイト** の2種別。マイルストーン（標準進捗）の計算はプロジェクト設定に従う。

---

## 3. 認証

### 3.1 ログイン
- Google アカウントで OAuth 2.0 ログイン
- 初回ログイン時、employees テーブルにレコードが存在しない場合は自動作成（role = `employee`）
- ログイン後は `/`（ダッシュボード）へリダイレクト

---

## 4. 機能要件

### 4.1 ダッシュボード（全ロール共通）

#### 個人進捗表示
- 参加しているプロジェクト一覧をタブ切り替えで表示
- 選択中プロジェクトのスキル達成数・総スキル数・達成率を表示
- 累計勤務時間（CSV取込データから集計）を表示
- フェーズ別の標準進捗率（青）と実績達成率（橙）を並べて表示
- カテゴリ別（接客 / 調理 / 管理 / その他）のスキル達成状況をレーダーチャートで表示

#### チームランキング
- 同じチーム内の社員スキル達成率をランキング表示
- 1〜3位にメダル表示
- 標準進捗率との差（GAP）を表示

#### 未読通知
- 自分のスキル認定申請に「認定」「差し戻し」が返ってきた未読件数をバナーで表示

#### 管理者向けバッジ
- manager/admin/ops_manager のみ：認定待ちのスキル申請件数バッジを表示
- admin/ops_manager のみ：審査待ちのチーム変更申請件数バッジを表示

---

### 4.2 スキルチェックリスト（全ロール）

#### 一覧表示
- プロジェクト選択（複数プロジェクト参加者はタブ切り替え）
- フェーズ別 → カテゴリ別にグループ化して表示
- 各スキルにステータスアイコンを表示
  - ○ 未申請
  - ⏳ 申請中（審査待ち）
  - ✓ 認定済み
  - ✗ 差し戻し
- カテゴリごとの達成数プログレスバーを表示

#### スキル申請
- 未申請スキルを選択して申請ダイアログを開く
- コメント（任意）を入力して申請
- 申請時の累計勤務時間を自動記録

#### 再申請
- 差し戻されたスキルを再申請（コメント変更可）

#### 申請履歴
- 全申請の履歴一覧（認定・差し戻し含む）を切り替えタブで表示
- ステータスアイコン、カテゴリバッジ、申請日時、認定者コメントを表示

#### マイルストーン進捗
- フェーズごとの目安時間に対する現在の累計時間を表示
- 自分が今どのフェーズにいるかを可視化

---

### 4.3 スキル認定（manager / admin / ops_manager）

#### 認定待ち一覧
- 担当チームのメンバーからの pending 申請を一覧表示
- 担当チームメンバーを優先表示（その他メンバーは区分けして表示）
- 申請日時、スキル名、カテゴリ、申請者コメントを表示

#### 認定・差し戻し
- 確認ダイアログで申請内容を確認
- コメント（任意）を入力して「認定する」または「差し戻す」
- 処理後、申請者へ未読通知を生成（is_read = false）

#### 認定履歴
- 自分（ログインユーザー）が過去に認定・差し戻した申請の履歴を表示
- 処理日時、対象スキル、申請者、コメントを表示
- 承認=緑・差し戻し=赤で色分け

#### チーム一覧
- 担当チームメンバーの進捗サマリー（認定数 / 申請中 / 標準進捗GAP）を表示

---

### 4.4 チーム管理（manager / admin / ops_manager）

#### チーム一覧（全ロール閲覧）
- チーム種別（店舗 / チーム）と名称を一覧表示
- メンバー・マネージャーをチップ表示（主担当 / 副担当の区別あり）

#### admin / ops_manager — 直接編集
- **チーム作成**: チーム名・種別・主担当マネージャーを指定して即時作成
- **チーム名変更**: インライン編集
- **チーム削除**: 確認ダイアログ後に削除
- **メンバー追加 / 削除**
- **マネージャー追加（主担当 / 副担当）/ 削除**
  - 主担当は1チームにつき1名のみ
  - 主担当の変更時は既存主担当を副担当に自動降格

#### manager — 申請フロー
- **チーム作成申請**: チーム名・種別・担当マネージャーをリクエスト
- **メンバー追加 / 削除申請**
- **マネージャー追加 / 削除申請**
- 申請一覧で自分の申請のステータス（審査中 / 承認 / 差し戻し）を確認
- 差し戻された申請は再申請ボタンで再送信

#### 申請ステータス通知
- 申請結果（承認 / 差し戻し）を未読バッジで通知
- 申請タブを開いた際に自動で既読処理（SECURITY DEFINER 関数経由）

#### 審査履歴（admin / ops_manager）
- 自分が過去に承認・差し戻した申請の履歴を表示
- 申請内容・申請日時・審査日時・コメントを表示

---

### 4.5 メンバー管理（admin / ops_manager）

- 全社員一覧表示（名前・雇用形態・入社年度・役割・スキル達成率）
- 役割変更（社員 / メイト / マネージャー / 運用管理者 / 開発者）
- プロフィール画像アップロード（Supabase Storage 経由）
- チームでフィルタリング表示

---

### 4.6 プロジェクト管理（admin / ops_manager）

#### プロジェクト
- プロジェクトの作成・編集・削除
- アクティブ / 非アクティブ切り替え（非アクティブは社員の選択肢から除外）

#### フェーズ
- プロジェクト内にフェーズを複数定義
- 各フェーズに名称・順序（order_index）・目標累計時間（end_hours）を設定
- フェーズ追加・削除

#### スキル割当
- スキルをプロジェクトに紐づけ（project_skills）
- 紐づけたスキルをフェーズに割り当て（project_phase_id）
- フェーズ・スキルの紐づけにより、社員ごとの標準進捗率を自動計算

#### 社員のプロジェクト参加管理
- 社員をプロジェクトに参加させる（employee_projects）

---

### 4.7 労働時間 CSV インポート（manager / admin / ops_manager）

- CSV ファイル形式：`email, work_date (YYYY-MM-DD), hours`
- ブラウザ上で CSV をパース・検証（メール存在確認・日付形式・時間数妥当性）
- プレビュー表示（エラー行をハイライト）
- 確認後に `work_hours` テーブルへ upsert（同一 employee_id + work_date は上書き）

---

### 4.8 view-as（代理表示機能）

- manager / admin / ops_manager が任意の社員として画面を閲覧
- 「○○さんとして見ています」バナーをヘッダーに表示
- ダッシュボード・スキルチェックリストの全情報を当該社員視点で表示
- チーム管理では代理申請が可能（申請者 = 対象社員として記録）
- 解除で自分の画面に戻る

---

## 5. データモデル

### 主要テーブル

| テーブル | 説明 |
|---------|------|
| `employees` | 社員マスタ（auth連携・役割・雇用形態） |
| `skills` | スキルマスタ（名称・カテゴリ・表示順） |
| `achievements` | スキル申請・認定履歴 |
| `work_hours` | 勤務時間実績（CSV取込） |
| `skill_projects` | スキル習得プロジェクト定義 |
| `project_phases` | プロジェクト内フェーズ定義（目標時間付き） |
| `project_skills` | プロジェクト×スキル×フェーズの紐づけ |
| `employee_projects` | 社員のプロジェクト参加 |
| `teams` | チーム定義（店舗 / プロジェクト） |
| `team_members` | チームメンバー |
| `team_managers` | チームマネージャー（主担当 / 副担当） |
| `team_change_requests` | チーム変更申請・審査記録 |

### スキル申請ステータス遷移

```
未申請 → pending（申請）→ certified（認定）
                        → rejected（差し戻し）→ pending（再申請）
```

### チーム変更申請ステータス遷移

```
pending（申請）→ approved（承認）
              → rejected（差し戻し）→ pending（再申請）
```

---

## 6. 標準進捗率の計算ロジック

1. 社員の参加プロジェクトのフェーズ一覧を取得
2. 各フェーズに属するスキル数を集計（`skillsByPhase`）
3. フェーズの `end_hours` からマイルストーンマップ（start / end 時間帯）を構築
4. 現在の累計勤務時間がどのフェーズに位置するかを判定
5. 現在フェーズ内での進捗率（0〜100%）を算出
6. フェーズ内に含まれるスキル数で加重して全体標準進捗率を計算

---

## 7. 通知仕様

| 通知種別 | トリガー | 表示場所 | 既読処理 |
|---------|---------|---------|---------|
| スキル認定結果 | certified / rejected に更新 | ダッシュボードバナー・ナビバッジ | スキル詳細確認時（is_read = true） |
| チーム変更申請結果 | approved / rejected に更新 | チーム管理・申請タブバッジ | 申請タブを開いた時（applicant_read_at） |
| 認定待ち件数 | pending achievements が存在 | ナビバッジ（manager以上） | 認定処理後に件数が減少 |
| チーム申請審査待ち件数 | pending team_change_requests が存在 | ナビバッジ（admin/ops_manager） | 承認・差し戻し後に件数が減少 |

---

## 8. 非機能要件

### 8.1 セキュリティ
- Supabase RLS によりテーブル行レベルの権限制御
- `SECURITY DEFINER` 関数で manager の RLS 制限を安全に回避（既読処理）
- view-as 機能は manager / admin / ops_manager のみ利用可能

### 8.2 パフォーマンス
- 各ページで `Promise.all` による並列クエリ実行
- 認証 → 社員情報取得 → プロジェクト選択 → その他全クエリ並列、という最小限の直列チェーン
- クライアントサイドのオプティミスティック更新で操作のレスポンスを即時化

### 8.3 対応環境
- モバイルファースト（スマートフォン縦持ち基準のレイアウト）
- PWA 非対応（ブラウザアプリとして動作）
